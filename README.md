# Задание 1

## 1.1. Восстановление данных за предыдущий день

Подходит **полное резервное копирование (Full Backup)** — раз в сутки, например, ночью.
Делается полный снимок базы, который хранится и при необходимости разворачивается целиком.
Это самый простой и надёжный способ, когда допустима потеря данных за один день.


## 1.2. Восстановление данных за час до поломки

Подходит **инкрементное резервное копирование** в связке с полным:

- Раз в сутки делается **полный бэкап**
- Каждый час делается **инкрементный бэкап** — только изменения с момента последнего бэкапа

При восстановлении: сначала накатывается полный бэкап, затем поочерёдно все инкрементные
вплоть до нужного момента. Для финансовых компаний также применяют **WAL-логи**
(журналы транзакций) — они позволяют восстановить состояние БД до конкретной минуты.

## 1.3.* Моментальное переключение при поломке

Да, такой кейс возможен — это называется **Failover (автоматическое переключение)**.
Реализуется через:

- **Master-Slave репликацию** — slave постоянно получает данные с master и при его
  падении автоматически становится новым master
- **Кластеризацию** (например, PostgreSQL Patroni, MySQL Group Replication) — система
  сама определяет сбой и переключает трафик на резервный узел без участия администратора

Данные при этом теряются минимально — только те транзакции, которые не успели
реплицироваться в момент сбоя.

# Задание 2

## 2.1. Резервное копирование

Создание дампа базы данных `mydb` в бинарном формате:

```bash
pg_dump -U postgres -F c mydb > mydb_backup.dump
```

**Параметры:**
- `-U postgres` — пользователь БД
- `-F c` — формат Custom (бинарный, сжатый)
- `mydb` — имя базы данных
- `> mydb_backup.dump` — файл для сохранения

## 2.1. Восстановление

Сначала создаём пустую БД, затем восстанавливаем из дампа:

```bash
createdb -U postgres mydb_restored
pg_restore -U postgres -d mydb_restored mydb_backup.dump
```

Если дамп сохранён в SQL-формате — восстановление через `psql`:

```bash
psql -U postgres -d mydb_restored -f mydb_backup.sql
```

## 2.1.* Автоматизация

Да, возможна — через **cron** (планировщик задач Linux).

Создаём bash-скрипт `/scripts/pg_backup.sh`:

```bash
#!/bin/bash
TIMESTAMP=$(date +%F-%H-%M)
DB_NAME="mydb"
USER="postgres"
BACKUP_DIR="/backups/postgresql"

mkdir -p $BACKUP_DIR
pg_dump -U $USER $DB_NAME > $BACKUP_DIR/${DB_NAME}_${TIMESTAMP}.sql
```

Добавляем в crontab запуск каждую ночь в 02:00:

```bash
crontab -e
# Добавить строку:
0 2 * * * /scripts/pg_backup.sh
```

Теперь каждую ночь автоматически создаётся бэкап
с меткой времени в имени файла — например `mydb_2026-06-08-02-00.sql`.

# Задание 3

## 3.1. Инкрементное резервное копирование

Инкрементное копирование в MySQL реализуется через **Binary Log (binlog)** —
журнал всех изменений данных с момента последнего полного бэкапа.

### Шаг 1. Включить binary log в конфиге MySQL

```bash
sudo nano /etc/mysql/mysql.conf.d/mysqld.cnf
```

Добавить строки:

```ini
[mysqld]
server_id       = 1
log_bin         = /var/log/mysql/mysql-bin.log
expire_logs_days = 10
```

Перезапустить MySQL:

```bash
sudo service mysql restart
```

### Шаг 2. Сделать полный бэкап (отправная точка)

```bash
mysqldump -u root -p \
  --all-databases \
  --single-transaction \
  --flush-logs \
  --master-data=2 \
  > /backups/full_backup.sql
```

### Шаг 3. Инкрементный бэкап (копируем binlog-файлы)

Сбрасываем активный лог и копируем накопленные файлы:

```sql
-- В MySQL-консоли:
FLUSH BINARY LOGS;
```

```bash
cp /var/log/mysql/mysql-bin.000001 /backups/inc_$(date +%F_%H-%M).bin
```

### Шаг 4. Восстановление из инкрементного бэкапа

Сначала восстанавливаем полный бэкап, затем накатываем binlog:

```bash
# 1. Восстановить полный бэкап
mysql -u root -p < /backups/full_backup.sql
# 2. Применить инкрементные изменения из binlog
mysqlbinlog /backups/inc_2026-06-08_02-00.bin | mysql -u root -p
```

## 3.1.* Когда репликация выгоднее резервного копирования

| Ситуация | Бэкап | Репликация |
|---|---|---|
| Сервер упал — нужно быстро восстановиться | Долго: разворачивать бэкап | Мгновенно: slave становится master |
| Высокая нагрузка на чтение | Не помогает | Запросы распределяются на slave |
| Нужен живой дубль данных в реальном времени | Данные устаревшие | Данные актуальны |
| Случайно удалили данные (DROP TABLE) | Можно восстановить | Удаление реплицируется — не поможет |
| Долгосрочное хранение истории | Идеально | Не предназначена для этого |

**Вывод:** репликация выигрывает там, где важна **скорость переключения и доступность**,
а бэкап незаменим при **логических ошибках** (удаление данных, порча таблиц).
Идеальная стратегия — использовать **оба метода вместе**.
